# PIN Verification Feature

## Overview

HospiScanner requires a 6-digit PIN before displaying QR codes that contain sensitive patient information. The PIN is no longer derived from the DNI. Instead, the QR must include an explicit credential hash generated by HospiSafe Desktop.

## QR Contract

Sensitive medical QRs should include one of these SHA-256 hash fields:

- `pin_hash`
- `pinHash`
- `access_pin_hash`
- `accessPinHash`
- `verification_hash`
- `verificationHash`

For development fixtures only, the parser also accepts `pin`, `access_pin`, or `accessPin` and hashes it in memory. Production QRs should not include plaintext PINs.

## Flow

1. Scan QR code.
2. Parse JSON.
3. Detect sensitive data such as DNI, diagnosis, treatment, medications, or allergies.
4. Require a PIN hash for sensitive medical data.
5. Show the PIN dialog.
6. Hash the entered 6-digit PIN with SHA-256.
7. Compare the hash and reveal the report only when it matches.

## Example

```json
{
  "patient_id": "P-12345",
  "name": "Juan Perez",
  "DNI": "12345678",
  "diagnosis": "Observation",
  "bed": "A-101",
  "pin_hash": "481f6cc0518fbe4bfb0edb2a01d2c4bba956b630512252157ae1f410a459ccc1"
}
```

## Implementation

- `QRDataParser.parseQRData()`: Parses JSON, blocks sensitive reports without a PIN hash, and strips PIN fields from formatted display JSON.
- `QRDataParser.hashPin()`: Produces the SHA-256 hash used for verification.
- `QRDataParser.verifyPin()`: Validates a 6-digit numeric PIN against the expected hash.
- `ScannerViewModel.verifyPin()`: Opens the pending scan result only after a valid hash match.

## Security Notes

- Local history is stored through SQLCipher.
- App backups are disabled and backup rules exclude local app data.
- Copying medical reports to the clipboard requires confirmation.
- PIN validation is still local, so production deployments should pair this with audit logging, server-side policy, or short-lived signed QR payloads when possible.
